Summary

The monitor runs scheduled HTTP checks, records each result and calculates service-level summaries from the stored history.

My role and contribution

I designed and implemented the monitoring workflow: timer-triggered checks, storage, read endpoints, logging and deployment automation.

Status

Available as a public repository. Not presented here as a continuously running production service.

Technology

Python, Azure Functions, Azure Table Storage, GitHub Actions, Terraform basics.

Problem

A single request only shows whether a service responds now. Useful monitoring needs repeated checks, durable history and understandable outputs when something fails.

User workflow

To be documented — how an operator configures targets, reviews history and calls summary endpoints.

What I personally built

  • Timer-triggered checks with graceful HTTP and timeout error handling.
  • Azure Table Storage persistence for check history.
  • HTTP endpoints for uptime percentage, average response time and failure counts.
  • Structured logs, GitHub Actions deployment and Terraform basics.

Architecture or data flow

Scheduled collection is separated from read endpoints so monitoring can continue independently of dashboard or API requests. Each check is stored as a record so summaries can be recalculated from history.

To be documented — diagram of timer → storage → summary endpoints.

Engineering decisions

I separated scheduled collection from read endpoints and stored each check as a durable record rather than locking results into one presentation format.
