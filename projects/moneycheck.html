Summary

MoneyCheck evaluates a planned purchase with structured metrics before spending. Effective income is monthly earnings plus free or extra money (for example a gift). Deterministic TypeScript calculations produce the scores; AI chat is for educational explanations only.

My role and contribution

I product-directed and guided an AI-assisted build. I am the sole GitHub committer. Verified human contributions: ensuring onboarding asks for monthly amount, manually checking that AI chat works, and informal testing with a couple of people. This page does not claim independent authorship of core modules or unaided fluency across every code path.

Status

Public demo on Vercel (money-check-theta.vercel.app). No post-prototype maintenance claimed after the initial guided build.

Technology

Next.js, React, TypeScript, Supabase, PostgreSQL, OpenAI SDK, Vercel — as present in the repository and live deployment.

Problem

A purchase price alone rarely shows whether a decision is manageable. MoneyCheck is positioned for students and early-career users who want structured metrics before they spend (intent only — not a claim about live user volume).

User workflow

  • If not logged in, middleware sends the user to login.
  • If onboarding is incomplete, the account is not treated as ready.
  • Running a money check saves a record with an id; the result page loads that id for the signed-in user.
  • AI chat can explain results; it does not compute risk, regret, or scores.

What I personally built

  • Guided the AI-assisted implementation (Cursor for building, ChatGPT for planning, Google Stitch for UI).
  • Product checks: monthly-amount onboarding prompt; manual AI chat verification.
  • Informal testing with a couple of people.

Core application modules were generated under that guidance. I am still building unaided fluency of the full architecture.

Architecture or data flow

Auth-gated routes and a calc-versus-AI split exist in the codebase: TypeScript calculations produce structured outputs; AI explains those outputs. Migrations and RLS definitions exist in the repository. Live User A/B isolation and production RLS application are not claimed here.

Engineering decisions

Keeping deterministic calculations separate from AI explanation is visible in the code and is the most defensible technical pattern to highlight. Scores come from TypeScript, not from the model.

AI-assisted development

Required disclosure: Cursor (building), ChatGPT (planning), Google Stitch (UI). The README was AI-generated and is not treated as my independent engineering narrative. The build was largely AI-authored under human guidance.
